# AGENTS.md

本仓库用于开发一个植物大战僵尸风格的 2D 塔防项目，目标引擎为团结引擎 1.8.5。

## 项目基准

- 使用团结引擎 1.8.5 开发；不要擅自升级或降级编辑器版本。
- 默认以 2D 横向草坪塔防为核心体验：格子放置、阳光经济、植物攻击、僵尸推进、波次管理、关卡胜负。
- 项目可以借鉴经典塔防玩法结构，但不要直接使用、复制或导入《植物大战僵尸》的原始美术、音频、商标、角色名或关卡资源。
- 优先面向可运行原型，后续再扩展平台适配、商业化和内容量。

## 推荐目录约定

初始化团结项目后，优先采用以下目录结构：

- `Assets/_Project/Scripts/`：项目 C# 脚本。
- `Assets/_Project/Scenes/`：主场景、测试场景和关卡场景。
- `Assets/_Project/Prefabs/`：植物、敌人、投射物、格子、UI 等预制体。
- `Assets/_Project/Art/`：项目自制或可授权使用的图片资源。
- `Assets/_Project/Audio/`：项目自制或可授权使用的音频资源。
- `Assets/_Project/Data/`：ScriptableObject 配置，例如植物、僵尸、关卡、波次。
- `Assets/_Project/UI/`：UI 预制体、图集和界面资源。
- `Assets/ThirdParty/`：第三方插件或素材，并保留许可证说明。

## 代码规范

- 使用 C#，遵循团结/Unity 常见命名习惯：类型和公共成员使用 `PascalCase`，私有字段使用 `camelCase` 或 `_camelCase`，与现有代码保持一致。
- 游戏运行逻辑优先拆成清晰的小模块，例如 `GridSystem`、`PlantPlacementSystem`、`SunSystem`、`WaveSystem`、`CombatSystem`。
- 数据配置优先使用 ScriptableObject，不把大量植物、僵尸、关卡数值硬编码进 MonoBehaviour。
- 避免在 `Update()` 中做不必要的全局查找或频繁分配；需要大量生成/销毁的对象优先使用对象池。
- MonoBehaviour 负责连接场景对象和生命周期；纯规则、计算和状态流转尽量放进可测试的普通 C# 类。
- 不要引入大型框架或复杂架构，除非它明显降低当前项目复杂度。

## 场景和资源规则

- 主场景应保持轻量，核心玩法对象尽量通过预制体和配置驱动。
- 植物、僵尸、投射物、阳光、格子等高频对象应制作成预制体。
- 美术资源导入后注意 Sprite 设置、Pixels Per Unit、图集打包和透明边缘。
- UI 使用团结/Unity UI 或项目已有 UI 框架；不要混用多套 UI 技术，除非有明确理由。
- 第三方素材必须确认授权；不要提交来源不明的商业游戏素材。

## 测试和验证

- 每次修改核心玩法后，至少在编辑器中验证一个最小闭环：开始关卡、放置植物、生成阳光、僵尸出现、攻击命中、胜负条件。
- 如果后续加入自动化测试，优先覆盖纯 C# 规则模块，例如格子坐标、波次调度、伤害计算、冷却计时。
- 提交前检查 Console，确保没有新增错误日志、缺失引用或脚本编译错误。

## 版本控制注意事项

- 提交团结项目时应保留 `Assets/`、`Packages/`、`ProjectSettings/` 等关键目录。
- 不要提交 `Library/`、`Temp/`、`Obj/`、`Logs/`、`UserSettings/` 等本地生成目录。
- 修改 `.meta` 文件要谨慎；移动或重命名资源时尽量使用编辑器操作，保持引用稳定。
- 主分支只保留稳定基线；后续功能、修复和实验默认在独立分支或 git worktree 中开发。
- 本地 worktree 优先放在 `.worktrees/`，该目录不得提交进仓库。

## 协作约定

- 修改前先理解现有场景、预制体和配置文件，不要随意重命名公共资源路径。
- 保持每次改动聚焦在一个明确目标上，避免顺手重构无关系统。
- 若发现团结版本、包版本或平台设置与本文件冲突，先说明风险，再决定是否调整。
